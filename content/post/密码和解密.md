# AES

>Rijindael算法可以支持的区块和密钥长度可以是32的整数倍 以128为下限256位为上限，AES的区块长度固定为128 比特，密钥长度则可以是128，192或256比特

>AES加密过程是在一个4×4的字节矩阵上运作，这个矩阵又称为“状态（state）”，其初值就是一个明文区块（矩阵中一个元素大小就是明文区块中的一个Byte）。（Rijndael加密法因支持更大的区块，其矩阵行数可视情况增加）加密时，各轮AES加密循环（除最后一轮外）均包含4个步骤：

> 1. AddRoundKey — 矩阵中的每一个字节都与该次轮秘钥（round key）做XOR运算；每个子密钥由密钥生成方案产生。

>2. SubBytes — 通过个非线性的替换函数，用查找表的方式把每个字节替换成对应的字节。

>3. ShiftRows — 将矩阵中的每个横列进行循环式移位。

> 4. MixColumns — 为了充分混合矩阵中各个直行的操作。这个步骤使用线性转换来混合每列的四个字节。

# RSA算法

>加密公式 ``` 密文=明文^E mod N``` 

>对于一个密文 只需要知道e和n就可以了，所以公钥就是包含的这两个信息

>解密公示 ``` 明文 = 密文^D mod N``` 同时包含D和N的就是私钥了，但是因为N也包含在公钥中，所以只包含D的也可以称为私钥                                                                                                                 

>N 是两个很大的质数的p、q的乘积。如果明文大于N则再解密时无法得到正确的明文

>L 是p-1和q-1的最大公约数

>E 比1大 比L小且和E的最大公约数是1 

> D 满足E x D mod L = 1  保证对密文进行解密可以得到原文

>中间人攻击 A要将信息发送给B，但是C是中间人，在A向B索要公钥时(B解密),C将B下发的公钥拦截，将C自己的公钥发送给A，这时候A使用公钥（C的）加密发送"hello world"给B，消息被C拦截，因为是用C的公钥加密的，所以C可以用私钥（C自己的）进行解密，将解密后的密文用B的公钥（上一步拦截的）加密（加密的内容可任意篡改），发送给B，造成攻击

# 证书生成

> A如果要将公钥经签名后发送给B 需要：

>将公钥发送给受信任机构，受信任机构在确认没有问题后用私钥对A的公钥签名

>将机构的公钥和签名后的A的公钥发送给B

>B用机构的公钥验证签名没有问题后获取A的公钥

# 公钥基础设施（pki）

> 用户：想要注册公钥的人

>认证机构（CA）：生成秘钥对（也可以用户生成），注册公钥时对本人身份进行认证，生成并颁发证书，作废证书

>仓库：保存证书的数据库，也叫做证书目录，pki用户在需要的时候可以从中获取证书

>只有在查询认证机构最新的CRL，确认该证书是有效的并且该证书有合法的认证机构签名和有效期才说明证书一定有效

>一个认证机构公钥可以由另一个机构进行验证，这样就会形成层级，最上层的机构被称为根CA，验证的时候从根证书自上而下进行验证

## tls握手协议
![](/img/tls.jpeg)
> 握手协议：用于客户端和服务端商议密码协议，服务端将证书发送给客户端。生成共享密钥的时候可能被窃听，所以使用公钥或者Diffeie-hellman交换密钥，如果不使用客户端认证则不需要客户端发送证书，否则需要。预备主密码（48字节）由客户端生成，客户端和服务器使用服务器公钥加密生成相同的主密码。客户端在向服务端发送客户端证书时需要计算主密码和握手协议中传送的消息的散列值并加上自己的数字签名后发送给服务器。在确定后进行密码切换，先是客户端发送切换密码，然后试用对应的密码协议发送finished，然后服务端同样先发切换密码协议然后发送finished，注意切换密码协议后由记录协议进行加密然后发送消息

> 密码规格变更协议：在商议密码套件后，释放变更密码协议的信号进行密码变更

>警告协议：协议中途发生错误，例如消息无法解密

>应用数据协议：将tls上面的应用数据传达给通讯对象 

> 记录协议：将消息分成若干个片段，用握手协议阶段确定的算法进行压缩。经过压缩的片段加上消息认证码，通过附件的消息认证码的mac值，防止篡改。为了防止重放，计算消息验证码时加上片段编号。单向散列函数和共享秘钥需要和通信对象商议，然后使用对称密码将消息验证码和压缩的片段进行加密，最后组成报文进行发送
